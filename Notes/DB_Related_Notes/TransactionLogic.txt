🧠 TRANSACTION SYSTEM NOTES (FINTECH LEVEL)
🔑 1. Core Idea

Money transfer = Transaction + Ledger entries

Transaction → event (kya hua)
Ledger → actual money movement (truth)
🔄 2. Complete Flow (Must Memorize)
1. Validate Request
2. Idempotency Check
3. Account Validation
4. Balance Check
5. Start DB Transaction
6. Create Transaction (PENDING)
7. Create DEBIT Entry
8. Create CREDIT Entry
9. Mark Transaction COMPLETED
10. Commit Transaction
11. Send Email

👉 Short memory:

VALIDATE → IDEMPOTENCY → ACCOUNT → BALANCE → TXN → DEBIT → CREDIT → COMPLETE → COMMIT → EMAIL

🧱 3. Data Flow Architecture
User Request
   ↓
Transaction Collection
   ↓
Ledger Collection (2 entries)
   ↓
Account Balance (derived)

👉 Rule:

❌ Balance store mat karo
✅ Balance = ledger se calculate karo

⚙️ 4. Important Concepts
🔐 Idempotency

👉 Same request 2 baar aaye toh duplicate payment na ho

const existingTxn = await transactionModel.findOne({ idempotencyKey })
Cases:
COMPLETED → return success
PENDING → wait / inform
FAILED → retry allow
REVERSED → retry allow
⚖️ Double Entry System

Har transaction me:

Sender → DEBIT
Receiver → CREDIT

👉 Isse system balanced rehta hai

💰 Balance Check
const balance = await account.getBalance()

👉 Rule:

Before debit → always check balance

🔄 MongoDB Transaction (Atomicity)
const session = await mongoose.startSession()
session.startTransaction()

👉 Ensure:

either sab kuch hoga
ya kuch bhi nahi hoga
🚨 5. Critical Bugs (Must Remember)
❌ 1. Missing await
const txn = transactionModel.findOne(...)

👉 Wrong

const txn = await transactionModel.findOne(...)

👉 Correct

❌ 2. No Try-Catch in Transaction

👉 Risk:

DEBIT ho gaya
CREDIT fail ❌
✅ Fix:
try {
   await session.commitTransaction()
} catch (err) {
   await session.abortTransaction()
} finally {
   session.endSession()
}
❌ 3. Race Condition (Double Spending)

👉 2 requests same time:

dono balance pass
double debit
💡 Solution:
DB locking
versioning
atomic updates
❌ 4. Status Mismatch

👉 "REVERSE" vs "REVERSED"

👉 Always keep enum consistent

🧠 6. Status Lifecycle
PENDING → COMPLETED
        → FAILED
        → REVERSED

👉 Rule:

Start = PENDING
End = COMPLETED / FAILED
📧 7. Email Strategy

👉 Always after commit

COMMIT SUCCESS → THEN EMAIL

❌ Email fail ≠ transaction fail

🧠 8. Perfect Backend Flow (Final Form)
TRY:
  validate input
  check idempotency
  check account status
  check balance

  start session

  create txn (PENDING)
  create debit
  create credit

  mark txn COMPLETED

  commit

CATCH:
  abort transaction

FINALLY:
  end session

SEND EMAIL
